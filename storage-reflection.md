My application allows users to upload and view photos in an S3 bucket even though S3 objects are private by default by using pre-signed URLs.

A pre-signed URL is a temporary, secure link generated by the backend. It includes permission to perform a specific action (such as uploading or downloading a file) for a limited amount of time. Because of this, the S3 bucket itself does not need to be made public.

When a user uploads a photo, the application generates a pre-signed URL that allows the user’s browser to upload the image directly to S3. S3 accepts the request because the URL is signed with valid credentials and has not expired.

When a user wants to view a photo, the application generates another pre-signed URL that allows temporary access to that specific object. The browser uses this URL to retrieve and display the image, even though the object is still private in S3.


The two Global Secondary Indexes (GSIs) created in the DynamoDB table are important because DynamoDB is optimized for specific query patterns. GSIs allow the application to query data in ways that are not possible using only the primary key.


The user-photos-index uses Hash key: user_id and Range key: uploaded_at

This index allows the application to retrieve all photos uploaded by a specific user. This is useful for displaying a user’s photos on their profile page.

The user_id groups all photos belonging to a user, and uploaded_at allows the results to be sorted by time. This makes it easy to show photos in chronological order, such as most recent first.

The feed-index uses Hash key: feed_key and Range key: uploaded_at

This index is used for displaying photos in the main feed. The feed_key determines which photos should appear in the feed (for example, public photos). The uploaded_at attribute allows the feed to be sorted by time, so newer photos appear first.


Both GSIs are designed based on how the application needs to read data One focuses on user-specific queries (profile view) and the other focuses on feed-based queries (home page)

Using uploaded_at as the range key in both indexes allows efficient sorting by time, which is important for both user profiles and feeds.

Overall, these GSIs improve performance and make the application scalable by avoiding full table scans and enabling efficient queries.
